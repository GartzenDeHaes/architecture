Background

As every C# programmer knows, the language has many powerful features. However this power brings with it complexity, which in turn can make code more bug-prone and harder to read and maintain. The goal of this guide is to help reduce this complexity by describing in detail the dos and don'ts of writing C# code. These rules exist to keep the code base manageable while still allowing coders to use C# language features productively.

Style, also known as readability, is what we call the conventions that govern our C# code. The term Style is a bit of a misnomer, since these conventions cover far more than just source file formatting.

One way in which we keep the code base manageable is by enforcing consistency. It is very important that any programmer be able to look at another's code and quickly understand it. Maintaining a uniform style and following conventions means that we can more easily use "pattern-matching" to infer what various symbols are and what invariants are true about them. Creating common, required idioms and patterns makes code much easier to understand. In some cases there might be good arguments for changing certain style rules, but we nonetheless keep things as they are in order to preserve consistency.

Design

General Goals

▽ You should choose a specific design objective for your project such as correctness, performance, or maintainability. Some design objectives are mutually exclusive, so be realistic. Correctness and maintainability are appropriate for most DOR systems.

Keep It Simple

▽ The application should be designed as simply as possible, unless added complexity makes the application easier to use, easier to maintain, or more reliable. Simplicity can be measured by the amount of code, the number of components, or by the amount of configuration required.

Open Standards

▽ Where ever practical, the system should use open standards such as ECMA-334 (C#) and ISO SQL-2003. Non standard, proprietary language and database features should be avoided.

Financial Systems

▽ All important data in financial systems should be fully auditable. This means that the original data entered into the system can be reconstructed along with all subsequent changes to the data. To do this, record all updates in a jounal table and design the database so that records are never deleted.

User Activity Logging

▽ To the extent practical, user activity in the system should be logged both for debugging and audit purposes.

Security Logging

▶ Some security events are required to be logged in syslog format and both stored locally and transmitted to a central syslog server.

The following events will be syslogged:

  • User account creation.
  • User permissions changes.
  • Help desk access to user information.
  • Help desk changes to user information.
  • User logon with a logon ID and password.
  • User logon failure.
  • Logon ID retreival (forgot logon).
  • Password reset.
  • Access to screens, features, or pages that require checking security permissions.
  • Denied access to screens, features, or pages that require checking security permissions.
  • Application specific audit and security events.

Web Application Organization

▶ Server-side code should be logically organized into: presentation or views, view models, controllers, an object model, and a data factory. Client-side JavaScript should be limited as much as possible. (TODO: needs disucssion.)

Data Layer Design Pattern

▶ In all cases, the data tier should use a factory pattern and support transactions across calls. Factory methods should take intrinsic types as arguments and return Data Transfer Objects (DTO's).

Some of the most common data access designs are:

  • Inline the stored procedure calls where ever needed.
  • Inline the stored procedure calls in the object model.
  • Create self loading data objects and collections.
  • Create a data factory class that makes all stored procedure calls and returns business objects.

Our choice is to use a data factory. The data factory should be implemented with an interface so that a mock factory class can be used for unit testing. The methods should return either a single object, or an IList of objects.

The DOR.Core project contains a common base class called MsSqlDataAccess to simplify data factory creation. Partial classes should be used logically partition the data factory.

Scoping

Namespaces

▶ Namespaces in c# files are required and should be choosen based on the name based on the project, and possibly its path. Fully qualified class names can be used in the code instead of using a using directive when many classes from many namepaces are used. Otherwise, use the using directive to simply the code.
link

Definition: Namespaces subdivide the global scope into distinct, named scopes, and so are useful for preventing name collisions in the global scope.

Pros:

Namespaces provide a (hierarchical) axis of naming, in addition to the (also hierarchical) name axis provided by classes.

For example, if two different projects have a class Foo in the global scope, these symbols may collide at compile time or at runtime. If each project places their code in a namespace, project1.Foo and project2.Foo are now distinct symbols that do not collide.

Cons:

Namespaces can be confusing, because they provide an additional (hierarchical) axis of naming, in addition to the (also hierarchical) name axis provided by classes.

Decision:

Use namespaces according to the policy described below.

Namespaces

  • Namespaces are required in .cs files to avoid runtime naming conflicts: 
    namespace Project1
{
	// The content of a namespace is indented
	
	// Commonly used tokens.
	enum Status 
	{ 
		Unused, 
		EOF, 
		Error 
	}  

	public class Filer 
	{ 
		Status _state;

		public bool AtEof() 
		{ 
			return _state == Status.EOF; // Uses our namespace's EOF.
		} 
	}
}

Nested Classes

▶ Although you may use public nested classes when they are part of an interface, consider a namespace to keep declarations out of the global scope.
link

Definition: A class can define another class within it; this is also called a member class. 

class Foo 
{
	// Bar is a member class, nested within Foo.
	private class Bar 
	{
		...
	}

}

Pros: This is useful when the nested (or member) class is only used by the enclosing class; making it a member puts it in the enclosing class scope rather than polluting the outer scope with the class name.

Decision: Do not make nested classes public unless they are actually part of the interface, e.g., a class that holds a set of options for some method.

Method Parameter Ordering

▶ When defining a method, parameter order is: inputs, then outputs and references.
link

Parameters to C# methods are either input to the function, output from the function, or both. Output and input/output parameters use ref or out keywords. When ordering function parameters, put all input-only parameters before any output parameters. In particular, do not add new parameters to the end of the function just because they are new; place new input-only parameters before the output parameters.

This is not a hard-and-fast rule. Parameters that are both input and output (often classes/structs) muddy the waters, and, as always, consistency with related functions may require you to bend the rule.

Local Variables

▶ Place a function's variables in the narrowest scope possible.

C# allows you to declare variables anywhere in a function. We encourage you to declare them in as local a scope as possible, and as close to the first use as possible. This makes it easier for the reader to find the declaration and see what type the variable is and what it was initialized to. In particular, initialization should be used instead of declaration and assignment, e.g. 

int i;
i = f();      // Bad -- initialization separate from declaration.
int j = g();  // Good -- declaration has initialization.

Early Return from Methods

▶ Use early return in methods to reduce nesting depth and simplfy code.

Many computer science courses teach a principle from functional theory that functions and methods should only have one exit or return point. The intent is to make the function easier understand. While this may be true for mathmatical functions, in computer programs it cause the programer to create an additional level of nesting for each exit case. This quickly makes the program difficult to read and difficult to debug. 

public bool IsAtEof()
{
	if (! _isOpen)
	{
		return true;	// GOOD: reduce nesting through early exit.
	}
	
	int pos = RecalcFilePos(_file);
	return pos >= _fileLength;
}

Classes

Classes are the fundamental unit of code in C# and this section lists the main dos and don'ts you should follow when writing a class.

Doing Work in Constructors

▶ There is no need to limit the constructor to trivial initialization. However, an Init() method can be used if the class instances are pooled or reused.
link

Definition: It is possible to perform initialization in the body of the constructor.

Pros: Convenience in typing. No need to worry about whether the class has been initialized or not.

Cons: The problems with doing work in constructors are:

  • None -- The cons usually identified for initializion in constructor generally don't apply to C#.

Decision: If your object is pooled or reused, consider having an explicit Init() method. Otherwise, initilized the object in the constructor.

Structs vs. Classes

▶ Use a struct only for passive objects that carry data; everything else is a class.
link

The struct and class keywords behave almost identically in C#. We add our own semantic meanings to each keyword, so you should use the appropriate keyword for the data-type you're defining.

structs should be used for passive objects that carry data, and may have associated constants, but lack any functionality other than access/setting the data members. The accessing/setting of fields is done by directly accessing the fields rather than through method invocations. Methods should not provide behavior but should only be used to set up the data members, e.g., constructor, destructor, Initialize(), Reset(), Validate().

If more functionality is required, a class is more appropriate. If in doubt, make it a class.

Note that member variables in structs and classes have different naming rules.

Inheritance

▶ Composition is often more appropriate than inheritance.
link

Definition: When a sub-class inherits from a base class, it includes the definitions of all the data and operations that the parent base class defines. In practice, inheritance is used in three major ways in C#: implementation inheritance, in which actual code is inherited by the child, and interface inheritance, in which only method names are inherited, and abstract classes, which do both.

Pros: Implementation inheritance reduces code size by re-using the base class code as it specializes an existing type. Because inheritance is a compile-time declaration, you and the compiler can understand the operation and detect errors. Interface inheritance can be used to programmatically enforce that a class expose a particular API. Again, the compiler can detect errors, in this case, when a class does not define a necessary method of the API.

Cons: For implementation inheritance, because the code implementing a sub-class is spread between the base and the sub-class, it can be more difficult to understand an implementation. The sub-class cannot override functions that are not virtual, so the sub-class cannot change implementation. The base class may also define some data members, so that specifies physical layout of the base class.

Decision:

Do not overuse implementation inheritance. Composition is often more appropriate. Try to restrict use of inheritance to the "is-a" case: Bar subclasses Foo if it can reasonably be said that Bar "is a kind of" Foo.

Limit the use of protected to those member functions that might need to be accessed from subclasses. Note that data members must always be private.

Operator Overloading

▶ Do not overload operators except in rare, special circumstances.
link

Definition: A class can define that operators such as + and / operate on the class as if it were a built-in type.

Pros: Can make code appear more intuitive because a class will behave in the same way as built-in types (such as int). Overloaded operators are more playful names for functions that are less-colorfully named, such as Equals() or Add().

Cons: While operator overloading can make code more intuitive, it has several drawbacks:

  • It can fool our intuition into thinking that expensive operations are cheap, built-in operations.
  • It is much harder to find the call sites for overloaded operators. Searching for Equals() is much easier than searching for relevant invocations of ==.

Decision:

In general, do not overload operators. You can define functions like Equals() and CopyFrom() if you need them.

However, there may be rare cases where you need to overload an operator such as when implmenting a class that can be semantically considered a builtin type. Examples include the Date and Money classes from DOR.Utility.

Access Control

▶ Make all data members private, and provide access to them through accessor properties as needed. Typically a variable would be called _foo and the accessor property Foo.
link

See also Inheritance and Method Names.

Declaration Order

▶ Use the specified order of declarations within a class: private data members, protected data members, constructors, public properties, methods, and finally static methods.

Your class definition should start with its private section, followed by its protected section and then its public section. If any of these sections are empty, omit them.

Within each section, the declarations generally should be in the following order:

  • Data Members, including static data members
  • Constructors
  • Destructor
  • Properties
  • Methods
  • Static methods

Write Short Methods

▶ Prefer small and focused methods.

We recognize that long methods are sometimes appropriate, so no hard limit is placed on function's length. If a method exceeds about 40 lines, think about whether it can be broken up without harming the structure of the program.

Even if your long method works perfectly now, someone modifying it in a few months may add new behavior. This could result in bugs that are hard to find. Keeping your methods short and simple makes it easier for other people to read and modify your code.

You could find long and complicated methods when working with some code. Do not be intimidated by modifying existing code: if working with such a method proves to be difficult, you find that errors are hard to debug, or you want to use a piece of it in several different contexts, consider breaking up the function into smaller and more manageable pieces.

Other C# Features

Exceptions

▶ Exceptions should usually be handled by the top-level, or outer scope, of a call chain.
link

Pros:

  • Exceptions allow higher levels of an application to decide how to handle failures in deeply nested functions, without the obscure and error-prone bookkeeping of error codes.
  • This prevents unchecked exceptions from being handled by the operating system and terminating the process.
  • Top-level exception handling makes it easier to understand the program flow.

Cons:

  • It can be difficult to isolate problems when exceptions are thrown from deaply nested code, or from code that iterates over a large number of items.

Decision:

Exceptions should usually be handled by the top-level, or outer scope, of a call chain. Web applications can use Application_OnError.

LINQ

▶ Language Integrated Query (LINQ) will be used with the Entity Framework and whever else appropiate.
link

Pros:

  • LINQ greatly simplifies data handling across all data sources.
  • LINQ, along with the Entity Framework, can increase programmer productivity as high as perhaps 30%.
  • LINQ code can be parrellized with PLINQ in dot net 4.

Cons:

  • LINQ has two different syntaxes, neither of which is especially consistant with c#.
  • It's unlikely that anyone on our staff can debug language expression trees or Entity SQL.

Decision:

Use LINQ and cut development time.

Reflection

▶ We do not normally use relection.
link

Definition: Reflection allows a programmer to query the class, methods, and properties of an object at run time.

Pros:

It's useful for dynamically loading assemblies at runtime.

In rare circumstances, it is useful even outside of dynamic loading.

Cons: A query of type during run-time typically means a design problem. If you need to know the type of an object at runtime, that is often an indication that you should reconsider the design of your class.

Decision:

If you find yourself in need of writing code that behaves differently based on the class of an object, consider one of the alternatives to querying the type.

Virtual methods are the preferred way of executing different code paths depending on a specific subclass type. This puts the work within the object itself.

If the work belongs outside the object and instead in some processing code, consider a double-dispatch solution, such as the Visitor design pattern. This allows a facility outside the object itself to determine the type of class using the built-in type system.

If you think you truly cannot use those ideas, you may use reflection. But think twice about it. :-) Then think twice again. Do not hand-implement an reflection-like workaround. The arguments against reflection apply just as much to workarounds like class hierarchies with type tags.

Naming

The most important consistency rules are those that govern naming. The style of a name immediately informs us what sort of thing the named entity is: a type, a variable, a function, a constant, a macro, etc., without requiring us to search for the declaration of that entity. The pattern-matching engine in our brains relies a great deal on these naming rules.

Naming rules are pretty arbitrary, but we feel that consistency is more important than individual preferences in this area, so regardless of whether you find them sensible or not, the rules are the rules.

General Naming Rules

▶ Function names, variable names, and filenames should be descriptive; eschew abbreviation. Types and variables should be nouns, while functions should be "command" verbs.
link

How to Name

Give as descriptive a name as possible, within reason. Do not worry about saving horizontal space as it is far more important to make your code immediately understandable by a new reader. Examples of well-chosen names: 

int numErrors;                  // Good.
int numCompletedConnections;   // Good.

Poorly-chosen names use ambiguous abbreviations or arbitrary characters that do not convey meaning: 

int n;                           // Bad - meaningless.
int nerr;                        // Bad - ambiguous abbreviation.
int nCompConns;                // Bad - ambiguous abbreviation.

Type and variable names should typically be nouns: e.g., fileOpener, numErrors.

Function names should typically be imperative (that is they should be commands): e.g., OpenFile(), SetNumErrors(). There is an exception for properties, which, described more completely in Function Names, should be named the same as the variable they access.

Abbreviations

Do not use abbreviations unless they are extremely well known outside your project. For example: 

// Good
// These show proper names with no abbreviations.
int numDnsConnections;  // Most people know what "DNS" stands for.
int priceCountReader;   // OK, price count. Makes sense.
 
// Bad!
// Abbreviations can be confusing or ambiguous outside a small group.
int wgcConnections;  // Only your group knows what this stands for.
int pcReader;        // Lots of things can be abbreviated "pc".

Never abbreviate by leaving out letters: 

int errorCount;  // Good.
 
int errorCnt;    // Bad.

File Names

▶ Filenames should be camel case, should not include underscores (_) or dashes (-), and be named after the class contained in the file. Each file usually contains a single class.
link

Examples of acceptable file names:

MyUsefulClass.cs

In general, make your filenames very specific. For example, use HttpServerLogs.cs rather than Logs.cs. A very common case is to have a file called FooBar.cs defining a class called FooBar.

Type Names

▶ Type names start with a capital letter and have a capital letter for each new word, with no underscores: MyExcitingClass, MyExcitingEnum.
link

The names of all types — classes, structs, typedefs, and enums — have the same naming convention. Type names should start with a capital letter and have a capital letter for each new word. No underscores. For example: 

// classes and structs
class UrlTable { ...
class UrlTableTester { ...
struct UrlTableProperties { ...

// enums
enum UrlTableError { ...

Variable Names

▶ Variable names start with lowercase and use uppercase for the first letter of each word. Class member variables have a leading "_". For instance: myExcitingLocalVariable, _myExcitingMemberVariable.
link

Common Variable names

For example: 

string tableName;  // OK
string tablename;   // OK - all lowercase.
 
string table_name;   // Bad - uses underscore.

Class Data Members

Data members (also called instance variables or member variables) start with "_", with the first word lowercase and the following words camel cased. 

string _tableName;  // OK.
string tablename;   // Bad -- no leading _.

Struct Variables

Data members in structs should be named like regular variables without the leading "_" that data members in classes have. 

struct UrlTableProperties {
  string name;
  int numEntries;
}

See Structs vs. Classes for a discussion of when to use a struct versus a class.

Constant Names

▶ Constants and enum members should be camel cased: DaysInAWeek.

All compile-time constants, whether they are declared locally or as part of a class, follow a different naming convention from other variables. Use camel case. 

const int DaysInAWeek = 7;

Method Names

▶ Regular functions and properties are camel cased; propreties should match the name of the variable: MyExcitingFunction(), MyExcitingMethod(), int MyExcitingMemberVariable { get ...
link

Regular Functions

Functions should start with a capital letter and have a capital letter for each new word. No underscores: 

AddTableEntry()
DeleteUrl()

Properties

Properties should match the name of the variable they are getting and setting. This shows an excerpt of a class whose instance variable is _numEntries. 

class MyClass 
{
	private int _numEntries;
	
	public int NumEntries 
	{ 
		get { return _numEntries; }
		set { _numEntries = value; }
	}
	
	// Or use auto-implemented properties
	public int MoreEntries 
	{ 
		get;
		private set;
	}
	
}

Namespace Names

▶ Namespace names are camel cased and are based on project names and possibly the directory structure: namespace GoogleAwesomeProject.
link

See Namespaces for a discussion of namespaces and how to name them.

Enumeration Names

▶ Enumerators should be camel cased and singular: MyExcitingEnumValue.

The individual enumerators should be camel case. The enumeration name, UrlTableError, is a type, and therefore mixed case. Enumeration names should be singular, not plural. 

enum UrlTableError
{
	Ok = 0,
	ErrorOutOfMemory,
	ErrorMalformedInput
}

Comments

Though a pain to write, comments are absolutely vital to keeping our code readable. The following rules describe what you should comment and where. But remember: while comments are very important, the best code is self-documenting. Giving sensible names to types and variables is much better than using obscure names that you must then explain through comments. When writing your comments, write for your audience: the next contributor who will need to understand your code.

Comment Style

▶ Use either the // or /* */ syntax, as long as you are consistent.
link

You can use either the // or the /* */ syntax; however, // is much more common. Be consistent with how you comment and what style you use where.

File Comments

▶ If applicable, start each file with a copyright notice.
link

Legal Notice and Author Line

When applicable, every file should contain the following items, in order:

  • a copyright statement (for example, Copyright 2008 by The State of Washington.)
  • a license boilerplate. Choose the appropriate boilerplate for the license used by the project (for example, Apache 2.0, BSD, LGPL, GPL)
  • an author line to identify the original author of the file

If you make significant changes to a file that someone else originally wrote, add yourself to the author line. This can be very helpful when another contributor has questions about the file and needs to know whom to contact about it.

Class Comments

▶ Every class definition should have an accompanying comment that describes what it is for and how it should be used.
link 
///<summary> Iterates over the contents of a GargantuanTable.  Sample usage:
///		foreach (GargantuanTableIterator iter in table) 
///		{
///			Process(iter.key(), iter.value());
///		}
///<summary>
class GargantuanTableIterator 
{
  ...
}

If you have already described a class in detail in the comments at the top of your file feel free to simply state "See comment at top of file for a complete description", but be sure to have some sort of comment.

Document the synchronization assumptions the class makes, if any. If an instance of the class can be accessed by multiple threads, take extra care to document the rules and invariants surrounding multithreaded use.

Method Comments

▶ Declaration comments describe use of the method; comments at the definition of a method describe operation.
link

Method Declarations

Every method declaration should have comments immediately preceding it that describe what the method does and how to use it. These comments should be descriptive ("Opens the file") rather than imperative ("Open the file"); the comment describes the function, it does not tell the function what to do. In general, these comments do not describe how the function performs its task. Instead, that should be left to comments in the function definition.

Types of things to mention in comments at the function declaration:

  • What the inputs and outputs are.
  • Whether any of the arguments can be null.
  • If there are any performance implications of how a function is used.
  • If the function is re-entrant. What are its synchronization assumptions?

Here is an example: 

///<summary> Returns an iterator for this table.  The client's
/// must not use the iterator once the GargantuanTable object
/// on which the iterator was created has been cleared.
///
/// The iterator is initially positioned at the beginning of the table.
///
/// This method is equivalent to:
///    Iterator iter = table.NewIterator();
///    iter.Seek("");
///    return iter;
/// If you are going to immediately seek to another place in the
/// returned iterator, it will be faster to use NewIterator()
/// and avoid the extra seek.
///</summary>
Iterator GetIterator();

However, do not be unnecessarily verbose or state the completely obvious. Notice below that it is not necessary to say "returns false otherwise" because this is implied. 

// Returns true if the table cannot hold any more entries.
bool IsTableFull();

When commenting constructors and finalizers, remember that the person reading your code knows what constructors and finalizers are for, so comments that just say something like "frees resources" are not useful. Document what constructors do with their arguments (for example, if they take ownership of references), and what cleanup the finalizer does. If this is trivial, just skip the comment. It is quite common for finalizers not to have a comment.

Function Definitions

Each function definition should have a comment describing what the function does and anything tricky about how it does its job. For example, in the definition comment you might describe any coding tricks you use, give an overview of the steps you go through, or explain why you chose to implement the function in the way you did rather than using a viable alternative. For instance, you might mention why it must acquire a lock for the first half of the function but why it is not needed for the second half.

Variable Comments

▶ In general the actual name of the variable should be descriptive enough to give a good idea of what the variable is used for. In certain cases, more comments are required.
link

Class Data Members

Each class data member (also called an instance variable or member variable) can have a comment describing what it is used for. If the variable can take sentinel values with special meanings, such as null or -1, document this. For example: 

 // Keeps track of the total number of entries in the table.
 // Used to ensure we do not go over the limit. -1 means
 // that we don't yet know how many entries the table has.
 private int _numTotalEntries;

Implementation Comments

▶ In your implementation you should have comments in tricky, non-obvious, interesting, or important parts of your code. Explain why something is being done, not how.
link

Class Data Members

Tricky or complicated code blocks should have comments before them. Example: 

// Divide result by two, taking into account that x
// contains the carry from the add.
for (int i = 0; i < result.Size(); i++) 
{
	x = (x << 8) + result[i];
	result[i] = x >> 1;
	x &= 1;
}

Line Comments

Also, lines that are non-obvious should get a comment at the end of the line. These end-of-line comments should be separated from the code a tab. Example: 

// If we have enough memory, mmap the data portion too.
mmap_budget = max<int64>(0, mmap_budget - index_->length());
if (mmap_budget >= data_size_ && !MmapData(mmap_chunk_bytes, mlock))
  return;  // Error already logged.

Note that there are both comments that describe what the code is doing, and comments that mention that an error has already been logged when the function returns.

If you have several comments on subsequent lines, it can often be more readable to line them up: 

...
DoSomething();                  // Comment here so the comments line up.
DoSomethingElseThatIsLonger();  // Comment here so there is a tab between
                                // the code and the comment.
...

null, true/false, 1, 2, 3...

When you pass in null, boolean, or literal integer values to functions, you should consider adding a comment about what they are, or make your code self-documenting by using constants. For example, compare: 

bool success = CalculateSomething
	(
		interestingValue,
		10,
		false,
		NULL	// What are these arguments??
	);

versus: 

bool success = CalculateSomething
	(
		interestingValue,
		10,     // Default base value.
		false,  // Not the first time we're calling this.
		NULL	// No callback.
	);

Or alternatively, constants or self-describing variables: 

const int kDefaultBaseValue = 10;
const bool FIRST_TIME_CALLING = false;
Callback *nullCallback = null;
bool success = CalculateSomething
	(
		interestingValue,
		DEFAULT_BASE_VALUE,
		FIRST_TIME_CALLING,
		nullCallback
	);

Don'ts

Note that you should never describe the code itself. Assume that the person reading the code knows C# better than you do, even though he or she does not know what you are trying to do: 

// Now go through the b array and make sure that if i occurs,
// the next element is i+1.
...        // Geez.  What a useless comment.

Punctuation, Spelling and Grammar

▶ Pay attention to punctuation, spelling, and grammar; it is easier to read well-written comments than badly written ones.
link

Comments should usually be written as complete sentences with proper capitalization and periods at the end. Shorter comments, such as comments at the end of a line of code, can sometimes be less formal, but you should be consistent with your style. Complete sentences are more readable, and they provide some assurance that the comment is complete and not an unfinished thought.

Although it can be frustrating to have a code reviewer point out that you are using a comma when you should be using a semicolon, it is very important that source code maintain a high level of clarity and readability. Proper punctuation, spelling, and grammar help with that goal.

TODO Comments

▶ Use TODO comments for code that is temporary, a short-term solution, or good-enough but not perfect.
link

TODOs should include the string TODO in all caps, followed by your name, e-mail address, or other identifier in parentheses. A colon is optional. The main purpose is to have a consistent TODO format searchable by the person adding the comment (who can provide more details upon request). A TODO is not a commitment to provide the fix yourself. 

/// TODO(kl@gmail.com): Use a "*" here for concatenation operator.
/// TODO(Zeke) change this to use relations.

If your TODO is of the form "At a future date do something" make sure that you either include a very specific date ("Fix by November 2005") or a very specific event ("Remove this code when all clients can handle XML responses.").

Formatting

Coding style and formatting are pretty arbitrary, but a project is much easier to follow if everyone uses the same style. Individuals may not agree with every aspect of the formatting rules, and some of the rules may take some getting used to, but it is important that all project contributors follow the style rules so that they can all read and understand everyone's code easily.

Non-ASCII Characters

▶ Non-ASCII characters should be rare and must use UTF-8 formatting.
link

You shouldn't hard-code user-facing text in source, even English, so use of non-ASCII characters should be rare. However, in certain cases it is appropriate to include such words in your code. For example, if your code parses data files from foreign it may be appropriate to hard-code the non-ASCII string(s) used in those data files as delimiters. More commonly, unittest code (which does not need to be localized) might contain non-ASCII strings. In such cases, you should use UTF-8, since that is an encoding understood by most tools able to handle more than just ASCII. Hex encoding is also OK, and encouraged where it enhances readability — for example, "\xEF\xBB\xBF" is the Unicode zero-width no-break space character, which would be invisible if included in the source as straight UTF-8.

For ASPX pages, be sure to set the character and content encoding on the page. Otherwise, the character will show up as an empty square on some browsers.

Spaces vs. Tabs

▶ Use only tabs.
link

We use tabs for indentation -- do not use spaces. You should set your editor to emit tabs when you hit the tab key. This allows developers to set the tab size to thier own spacing preferences.

Method Calls

▶ Return type on the same line as function name, parameters on the same line if they fit.
link

Functions look like this: 

ReturnType FunctionName(Type parName1, Type parName2) 
{
	DoSomething();
	...
}

If you have too much text to fit on one line: 

	ReturnType ReallyLongFunctionName
	(
		Type parName1,
		Type parName2,
		Type parName3
	) 
	{
		DoSomething();
		...
	}

Some points to note:

  • The return type is always on the same line as the function name.
  • The open and close parenthesis are always either both on the same line as the function name, or on lines by themselves.
  • There is never a space between the function name and the open parenthesis.
  • The open curly brace is always on a line by itself.
  • The close curly brace is always on the last line by itself.
  • All parameters should be named, with identical names in the declaration and implementation.
  • All parameters should be aligned if possible.
  • Default indentation is one space.
  • Wrapped parameters have a one tab indent.

Conditionals

▶ Prefer no spaces inside parentheses. The else keyword belongs on a new line.
link

There are two acceptable formats for a basic conditional statement. One includes spaces between the parentheses and the condition, and one does not.

The most common form is without spaces. Either is fine, but be consistent. If you are modifying a file, use the format that is already present. If you are writing new code, use the format that the other files in that directory or project use. If in doubt and you have no personal preference, do not add the spaces. 

if (condition) 
{
	...	// TAB
} 
else
{
	// else uses consistant block style
	...
}

If you prefer you may add spaces inside the parentheses: 

if ( condition ) 
{
		// spaces inside parentheses
	...	// TAB
} 
else 
{
	// else uses consistant block style
	...
}

Note that in all cases you must have a space between the if and the open parenthesis. Curly braces should always be used for statement blocks, even if there's only one line of code. 

if(condition)     // Bad - space missing after IF.
if (condition) {	// Bad - { on the same line as IF.
if(condition){		// Doubly bad.
 
if (condition)	// Good - proper space after IF { on a line by itself.
{

Loops and Switch Statements

▶ Switch statements must use braces for blocks. Empty loop bodies must use { }.
link

case blocks in switch statements generally should not have curly braces, unless the intent is to create a local scope. If you do include curly braces, they should be formated consistant with conditional code blocks.

Switch statements should always have a default case. If the default case should never execute, simply Debug.Assert or throw and exception: 

switch (var) 
{
case 0:  	// no indent
	...	// TAB indent
	break;

case 1:
	...
	break;

default:
	Debug.Assert(false);
	break;
}

Empty loop bodies should use { }. 

while (condition) 
{
	// Repeat test until it returns false.
}
for (int i = 0; i < kSomeNumber; ++i)
{
	// Good - empty body.
}
 
while (condition);  // Bad - looks like part of do/while loop.

Reference Expressions

▶ No spaces around the period.
link

The following are examples of correctly-formatted pointer and reference expressions: 

x = p.x;
y = r.y;

Note that:

  • There are no spaces around the period when accessing a member.

Return Values

▶ Do not surround the return expression with parentheses.
link

Return values should not have parentheses: 

return x;	// not return(x);

Preprocessor Directives

▶ Preprocessor directives should not be indented but should instead start at the beginning of the line.
link

Even when pre-processor directives are within the body of indented code, the directives should start at the beginning of the line. 

	// Good - directives at beginning of line
	if (lopsidedScore) 
	{
#if DISASTER_PENDING      // Correct -- Starts at beginning of line
		DropEverything();
#endif
		BackToNormal();
	}
 
// Bad - indented directives
if (lopsidedScore) 
{
    #if DISASTER_PENDING  // Wrong!  The "#if" should be at beginning of line
    DropEverything();
    #endif                // Wrong!  Do not indent "#endif"
    BackToNormal();
  }

Class Format

▶ Example:

The basic format for a class declaration (lacking the comments, see Class Comments for a discussion of what comments are needed) is: 

public class MyClass : OtherClass 
{
	private int _someVar;
	private int _someOtherVar;

	public MyClass()  // Regular 1 TAB indent.
	{
	}

	public MyClass(int var)
	{
	}

	public int SomeVar
	{
		get { return _someVar; }
		set { _someVar = value; }
	}

	// OR use auto-implemented properties
	public int SomeMore
	{
		get;
		private set;
	}

	public void SomeFunction()
	{
		// ...
	}

	public void SomeFunctionThatDoesNothing() 
	{
	}

	private bool SomeInternalFunction()
	{

	}

Things to note:

  • Any base class name should be on the same line as the subclass name.
  • For simple properties, the get and set statement blocks can be on a single line.
  • Methods should be seperated by a single blank line.
  • There should not be a new line after opening braces or between the last method and the closing brace for the class.
  • See Declaration Order for rules on ordering declarations within the class.

Namespace Formatting

▶ The contents of namespaces are indented.
link

Namespaces do add an extra level of indentation. For example, use:

Vertical Whitespace

▶ Use blank lines to group related code. Do not use more than blank line.
link

In general, blank lines should be used group, or seperate, logical chunks of code. Each blank line in your code should have specific meaning, such as seperating methods or the seperating groups of related operations in a method. Don't put more than one blank line between functions, don't start or end functions with a blank line, and be discriminating with your use of blank lines inside functions.

Don't start or end functions with blank lines: 

void Function() 
{

	// Unnecessary blank lines before and after

}

Don't start and end blocks with blank lines either: 

while (condition) 
{
	// Unnecessary blank line after
	...

	// Unnecessary blank line before
	...
}

	string fileContent = File.ReadAll(fileName);

	for (int pos = 0; pos < fileContent.Length; pos++)
	{
		...
	}

	// Processing complete, so do something else.
	return true;

SQL

The use of certain conventions can make SQL programming easier and more reliable.

Field Prefixes

▶ Field names should be prefixed with the abbreviated table name. Foriegn keys should use the name field name as in the foriegn table.

So, let's say you had a table named User. Fields in this table should be named like User_Name_Txt. Usr_Name_Txt would also be acceptable. If User_Name_Txt is referenced as a foriegn key in another table, the other table should use User_Name_Txt for its field name.

Field Suffixes

▶ Field names will use DOR's standard suffixes.
Number_Num
Text_Txt
Indicator_Ind
Flag_Ind
Money_Amt
Lookup code_Code
Identifier_Id

General Formatting

▶ Keywords should be capitalized. Field names should use the same case as in the table definition. Each clause keyword such as SELECT, FROM, WHERE, and GROUP BY should be on a line by itself. Each field in a field list should be its own line. Parentheses should be on a separate line, except for the IN value list. Variables and arguments should use the same name as the field they reference.

ASP.NET

ASP.NET supports a wide range of coding and structual styles and features.

Design of Standard POST Back Pages

▶ Pages using the traditional GET, POST, and response lifecycle should be designed with an MVC pattern. The ASPX file is the view, the ASPX.CS file is the controller, and the application object model is the model. The application object model and data access should be in a DLL so that it can be unit tested.

http://en.wikipedia.org/wiki/Model-view-controller

Use of AJAX

▶ AJAX pages are usually more complex to design and take longer to develop and test. They should be avoided, unless they provide significant business value to the customer.

Definition: Web pages can use javascript to asynchronously send and receive data from a web server. This feature can be used to create traditional thick-client N-tier applications that run in the web browser. In pratice, these applications tend to use client server designs.

Pros: AJAX can be used to create slick UI's that would be cumbersome to implement with traditional pages.

  • Desktop application features such as drag-drop, modal dialogs, and live grids can be implemented.
  • AJAX applications can have a "glitz" factor that could previously only be achived with flash or shockwave.
  • Some AJAX applications, such as Google maps, are much easier to use than equivalent traditional page model would be.

Cons: While AJAX has several benefits, it also has a number of costs.

  • For some UI designs, making asynchronous calls the the server can cause the user falsely assume that an action is complete. For example, let's say that the user drags a message from her inbox to the recycle bin. The UI indicates that the message has been moved, so the user closes the browser or clicks off the page. However the call to the server was not complete, so the operation was aborted. The next time she returns to the application, the message is back in her inbox.
  • AJAX UI's can take significantly more time to develop and test. More code is usually required to be written and JavaScript and CSS skills and debugging are required. Supporting multiple browsers and versions also require significantly more effort.
  • The way users typically use the back and forward buttons can cause unexpected results. For example, paging is often used to split up large result sets displayed in a grid. When paging through the results, users will often hit the back button to return to the previously display set of results. However, in an AJAX'ed grid this will cause the previous web page to be displayed.
  • AJAX designs have accessability issues both for people with disabilities and for people with limited bandwidth.
  • AJAX designs tend to place a heavier load on the server, since they often make frequent fine-grained calls. For example, one popular AJAX technique to provide auto-completion on input boxes (like the search on http://msdn.microsoft.com). Every character that's typed requires a call to the server to get the suggested matches.

Decision:

As a general rule, do not use asynchronous designs.

Control Names

▶ The two most common control naming conventions are to either name them like other variables, or to use a prefix to indicate the control type. Our pratice is to use a prefix to indicate the control type.

By default, ASP.NET puts the definition of the control in a sperate file from the implementation. Use of the type prefix provides a hint to the developer to look in the ASPX file for the definition. However due to the development of IDE's with auto-completion features, the type information provided by the prefix is not especially useful.

Code Behind vs Code In Front

▶ Pages and controls should use code behind. Exceptions: Use code in front if production updates need to be made without re-deploying the project, or if there is only trivial code associated with the page.

For example, an ASPX page may contain nothing but user controls and layout. In this case, use code behind or code in front at your option.

Services

Service Interfaces

▶ Services should return false on internal error and take intrinstic types, or struct's of intrinsic types, as arguments. Do NOT use XmlDocument, nor a string with XML content, as an argument or return. Do not allow service methods to throw exceptions.

The intent of these rules is to make the interface portable, but without introducting an excessive burden on the programmer.

Service Code File

▶ Put application logic in a DLL that is reference by the service, not directly in service source file.

To simplify unit testing, all of the application logic should be in a DLL.

Service Use & SOA

▶ Do not use services unless there is a business requirement for remote access.

Much like RPC, distributed objects, DCOM, and CORBA before it, SOA encourages developers to introduce unnecessary tiers into their applications. This kind of tiering can easily increase the code size by 20%, without adding any functionality for the user. It also violates design principle #1 -- keep it simple.

User Interface

Modes

▶ Pages and forms should only have one mode, or provide a visual context for the mode change. Another way to say the same thing is that all controls visible on a form must be valid for use at any time and forms do not dyanmically reconfigure their UI.

Consider the following UI:

Field A:
Field B:
Data:
 

This is supposed to work as follows:

  • The user enters a value in Field A, clicks search and the located record is displayed for edit.
  • The user enters a value in Field B, clicks search and the located record is displayed for edit.
  • The user changes Field A, Field B, and/or Data, and then updates the database by clicking Update.

This form has 3 modes: search on a, search on b, and update. The problems start when the user mixes these modes. For example, after a search the UI assumes that the user will either edit fields and click update, or change one of the search fields and click the correct search button. But what if the user edits Field A, edits Data, but then click Search A? The system must either discard the changes or update the current record before running the search.

You might think this problem can be solved by using a confirmation dialog. However, the even a simple form like this one has too many modes to be resolved this way. If the user changes Field A and then clicks Search B, should the system confirm the update or confirm whether the user meant to click Search A? To determine what the user really meant, you would need to string together a series of dialogs "Did you really mean to click Search A? Yes/No" and "If no, do you want to save your changes? Yes/No". Not exactly an user friendly solution.

▶ The easiest way to resolve mode problems is to create seperate screen for each mode. The UI above could be split into search and edit screens. The search results can have an edit link that to goes to the edit page. The edit page has update and cancel buttons. All controls are valid to be edited or click without any ambiguitity.

No Surprises

▶ The UI should be designed so that the user can anticipate what will happen when an element is clicked or selected.

Users have learned though experience what to expect when using various UI controls such as buttons, links, text boxes, and so on. If the application does something unexpected, the user has to stop and figure out what happened. One of the central principles of usibility is, "where an expert pauses, a beginner will fail." It's easy to avoid this problem by following common conventions for the function of controls.

  • Buttons operate (do something, calculate, submit)
  • Calandars may be active or passive, depending on the UI context
  • Check boxes are zero to many and passive (they do not initate any action)
  • Drop down lists are passive
  • Images are passive, unless they clearly appear to be a button or link
  • Labels are passive
  • Links navigate (change pages, go somewhere else)
  • Radio buttons are mutually exclusive and passive
  • Text boxes are passive

For passive controls, real-time validation is acceptable as long as it does not disrupt the user or reposition the focus.

Parting Words

Use common sense, be consistent, and KEEP IT SIMPLE.

If you are editing code, take a few minutes to look at the code around you and determine its style. If they use spaces around their if clauses, you should, too. If their comments have little boxes of stars around them, make your comments have little boxes of stars around them too.

The point of having style guidelines is to have a common vocabulary of coding so people can concentrate on what you are saying, rather than on how you are saying it. We present global style rules here so people know the vocabulary. But local style is also important. If code you add to a file looks drastically different from the existing code around it, the discontinuity throws readers out of their rhythm when they go to read it. Try to avoid this.
